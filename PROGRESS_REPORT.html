PharmBot v2 — รายงานความคืบหน้าโปรเจค

ระบบจำลองร้านขายยาชุมชนสำหรับฝึกซักประวัติและจ่ายยา · โปรเจคจบ นศภ. ปี 5

📅 ปรับปรุงล่าสุด: 25 มิถุนายน 2026 👥 ทีม 3 คน 🧪 Quasi-experimental: One-group pretest–posttest ⚙️ Vanilla JS + Firebase + Gemini

1 ภาพรวมโปรเจคและวัตถุประสงค์

บริบทและกรอบงานวิจัยสำหรับผู้ฟังที่ยังไม่ทราบรายละเอียด

PharmBot v2 เป็นเว็บแอปจำลองสถานการณ์ร้านขายยาชุมชน ให้นักศึกษาเภสัชศาสตร์ฝึก ซักประวัติ → จ่ายยา → ให้คำแนะนำ กับ "ผู้ป่วยจำลอง" ที่ขับเคลื่อนด้วย AI (Gemini) แล้วได้รับการประเมินและ feedback อัตโนมัติ ระบบ deploy แบบ static บน GitHub Pages โดยใช้ Firebase สำหรับ Authentication และ Firestore เป็นฐานข้อมูล

วัตถุประสงค์การวิจัยวิธีวัดสถานะการเก็บข้อมูล
1. ทักษะการซักประวัติ & การจ่ายยาคะแนนจาก AI evaluation (4 ด้าน)เก็บได้แล้ว
2. ความมั่นใจ (Self-efficacy)แบบสอบถาม Likert ก่อน–หลังยังไม่มีระบบ
3. ความพึงพอใจ / การใช้งานแบบสอบถาม SUS 10 ข้อยังไม่มีระบบ
ประเด็นสำคัญต่ออาจารย์: ตัวโปรแกรมพร้อมใช้งานแล้ว แต่กลไกเก็บข้อมูล วัตถุประสงค์ข้อ 2–3 (ความมั่นใจ + ความพึงพอใจ) ยังไม่ได้พัฒนา ซึ่งเป็นสิ่งที่ต้องทำก่อน deploy เก็บข้อมูลจริง

3 สิ่งที่พัฒนาเสร็จแล้ว

ฟังก์ชันที่ใช้งานได้จริงในระบบปัจจุบัน

✅ Flow การฝึก 4 ขั้นตอนซักประวัติ → จ่ายยา → counseling → ประเมินด้วย AI
✅ ผู้ป่วยจำลองด้วย AIpersona ตามเพศ/อายุ, ไม่เปิดเผยข้อมูลจนกว่าจะถูกถาม
✅ โหมดเสียง (Gemini Live)พูดโต้ตอบภาษาไทยแบบ real-time + transcript
✅ การประเมินอัตโนมัติให้คะแนน 4 ด้านเป็น JSON + feedback + checklist
✅ จับเวลาแบบ OSPEนับถอยหลัง 5 นาที + เตือน + กันรีเฟรชหลุดกลางคัน
✅ Admin panelจัดการเคส/ยา/กลุ่มโรค + ดูผลนักศึกษาทั้งหมด
✅ ระบบล็อกอินนักศึกษารหัส P-code + จำกัด 5 ครั้ง/คน/วัน
✅ คลังเคส 5 เคสเจ็บคอ, ท้องเสีย, ปวดหลัง, เชื้อราที่เท้า, ปวดหัว (refer)

4 ความคืบหน้าล่าสุด (รอบวันที่ 25 มิ.ย. 2026)

เน้นยกระดับ "คุณภาพ AI" เพื่อความน่าเชื่อถือของงานวิจัย

  • กันผู้ป่วยจำลอง "แต่งข้อมูลเอง" — เพิ่มกฎให้ AI ตอบ "ไม่แน่ใจ/จำไม่ได้" เมื่อถูกถามเรื่องที่ไม่ได้ระบุในเคส แทนการสุ่มสร้างอาการใหม่ → ลดความเสี่ยงที่ผู้ป่วยคนเดียวกันให้ข้อมูลไม่ตรงกันในแต่ละรอบ (กระทบความเที่ยงของการให้คะแนน)
  • ให้ AI ผู้ประเมิน "คิดก่อนให้คะแนน" — เพิ่ม field reasoning ใน output ให้วิเคราะห์ทีละหมวดโดยอ้างหลักฐานจาก transcript ก่อนสรุปเป็นคะแนน → คะแนนนิ่งขึ้นและอธิบายต่ออาจารย์ได้
  • ปรับ checklist ให้สอดคล้องกับน้ำหนักคะแนน — จากเดิม 30/20/50 ที่ขัดกับสูตรถ่วงน้ำหนัก เปลี่ยนเป็น 4 หมวดตรงกับ 4 domain แบบ 1:1 (25 / 15 / 40 / 20) ดูภาคผนวกสำหรับเหตุผลรองรับ
  • แยก AI model สำหรับผู้ป่วย ออกจาก model สำหรับการประเมิน (Phase 4) — เพิ่ม evalModel ในระบบ พร้อมสคริปต์ setup/update-models.js ที่รันอัพเกรดได้ทันที (รองรับค่าเดิมแบบ backward-compatible)
ทดสอบแล้ว: ทุกการแก้ไขผ่าน syntax check และ functional test (ผู้ป่วยมีกฎใหม่, eval มี reasoning + น้ำหนักถูกต้อง, การ fallback ของ model ทำงานและไม่ทำให้ระบบเดิมพัง)

5 สิ่งที่ยังขาด และข้อแนะนำ (เรียงตามความสำคัญ)

สิ่งที่ทีมควรวางแผนทำต่อ

ลำดับงานทำไมสำคัญสถานะ
สูงสุด Phase 1 — เครื่องมือเก็บข้อมูลวิจัย
แบบวัดความมั่นใจ (ก่อน–หลัง), SUS, แยก pretest/posttest, ส่งออก CSV		 เป็น blocker ของวัตถุประสงค์ข้อ 2–3 — ถ้าไม่มี จะเก็บข้อมูลวิจัยไม่ครบและวิเคราะห์ผลไม่ได้ ยังไม่เริ่ม
กลาง Phase 3 — เพิ่มเคสคลินิก
จาก 5 → 9–12 เคส + ให้อาจารย์เภสัชกรตรวจสอบพฤติกรรมผู้ป่วย		 ความหลากหลายของเคส + ความถูกต้องทางคลินิก (ต้องเผื่อเวลารออาจารย์รีวิว) 5/9–12
กลาง Phase 2 — Refactor โค้ด
สร้าง config.js รวมค่าคงที่ + แยกไฟล์ chat.js ที่ใหญ่เกินไป		 โค้ด chat.js ~1,300 บรรทัด ดูแล/ตรวจสอบยาก เสี่ยงพลาดเวลาแก้ใหญ่ ยังไม่เริ่ม
ต่อเนื่อง Phase 5 — Validation & Testing
แก้ Playwright tests ที่ fail + ทำ inter-rater reliability (AI vs ผู้เชี่ยวชาญ)		 ยืนยันความน่าเชื่อถือของคะแนน AI เพื่อใช้อ้างอิงในงานวิจัย บางส่วน fail
งานค้างเล็กน้อย: มีการแก้ chat.js (animation) และไฟล์ DEV_PLAN.md ที่ยังไม่ได้ commit เข้า git — แนะนำให้ commit เก็บก่อนเริ่มงานใหม่

6 แผนถัดไป (Roadmap ที่แนะนำ)

ลำดับที่ลดความเสี่ยงและปลดล็อกงานวิจัยเร็วที่สุด

  • ขั้นที่ 1 · Phase 1 — ระบบแบบสอบถาม ทำก่อน
    สร้าง /surveys, หน้า survey (confidence + SUS), เพิ่ม sessionNumber, ปุ่มส่งออก CSV — ปลดล็อกการเก็บข้อมูลวิจัยทั้งหมด
  • ขั้นที่ 2 · Phase 3 — เพิ่มเคส + Expert review
    เริ่มคู่ขนานได้ทันที เพราะต้องรอตารางอาจารย์ — มอบหมายให้เพื่อนช่วยเขียนเนื้อหาเคส
  • ขั้นที่ 3 · Pilot test ภายในทีม
    ทดลองใช้กันเองให้ครบ flow + ทดสอบ CSV export ก่อนเก็บข้อมูลจริง
  • ขั้นที่ 4 · Phase 2 & 5 — Refactor + Test + Reliability
    ปรับคุณภาพโค้ด, แก้ test, เทียบคะแนน AI กับผู้เชี่ยวชาญ
  • ขั้นที่ 5 · Deploy & เก็บข้อมูลจริง
    เปิดให้กลุ่มตัวอย่างใช้งานตามแผน pretest–posttest

7 ความเสี่ยงและข้อจำกัดที่ควรระบุในรายงาน

เพื่อความโปร่งใสและเตรียมตอบคำถามอาจารย์

  • ความเที่ยงตรงของการให้คะแนนด้วย AI — ต้องทำ inter-rater reliability เทียบกับผู้เชี่ยวชาญ เพื่อยืนยันว่าคะแนน AI ใช้อ้างอิงได้
  • ความถูกต้องของผู้ป่วยจำลอง — แม้เพิ่มกฎกัน AI แต่งข้อมูลแล้ว ยังควรให้อาจารย์เภสัชกรตรวจสอบพฤติกรรมผู้ป่วยในแต่ละเคส
  • ข้อจำกัดของ design — One-group pretest–posttest ไม่มีกลุ่มควบคุม ทำให้สรุปสาเหตุได้จำกัด ควรระบุเป็น limitation
  • ความแม่นของ transcript เสียง — โหมดเสียงอาศัยการถอดเสียงภาษาไทย ซึ่งอาจมีความคลาดเคลื่อนและกระทบการให้คะแนน
  • ขนาดและการเลือกกลุ่มตัวอย่าง — ควรระบุเกณฑ์คัดเข้า/ออกและจำนวนให้ชัดเจน

8 หัวข้อที่แนะนำให้มีในรายงานฉบับเต็ม / สไลด์นำเสนอ

โครงหัวข้อสำหรับเล่มวิทยานิพนธ์หรือการนำเสนอความก้าวหน้า

หัวข้อควรมีอะไรบ้าง
1. บทนำ / ที่มาและความสำคัญปัญหาการฝึกทักษะร้านยา, ช่องว่างที่ระบบนี้มาเติม, คำถามวิจัย
2. วัตถุประสงค์ & สมมติฐาน3 วัตถุประสงค์ (ทักษะ/ความมั่นใจ/ความพึงพอใจ) + ตัวแปรต้น–ตาม
3. ทบทวนวรรณกรรมOSCE/OSPE ในเภสัช, WWHAM, การใช้ simulated patient, AI ในการศึกษาเภสัช
4. ระเบียบวิธีวิจัยQuasi-experimental one-group pretest–posttest, กลุ่มตัวอย่าง, เครื่องมือ, การวิเคราะห์สถิติ
5. การออกแบบและพัฒนาระบบสถาปัตยกรรม (Vanilla JS + Firebase + Gemini), flow 4 ขั้นตอน, การออกแบบ prompt, การให้คะแนน
6. เครื่องมือวัดและการเก็บข้อมูลเกณฑ์ให้คะแนน (rubric/น้ำหนัก), แบบวัดความมั่นใจ, SUS, การ map ตัวแปรกับฐานข้อมูล
7. ความคืบหน้าการพัฒนาสิ่งที่เสร็จแล้ว, สิ่งที่กำลังทำ, ปัญหา/อุปสรรค (ส่วนนี้คือเนื้อห​าในรายงานฉบับนี้)
8. ผลเบื้องต้น / Pilot (ถ้ามี)ผลทดลองใช้ภายในทีม, ตัวอย่างคะแนนและ feedback
9. การประกันคุณภาพInter-rater reliability, expert review เคส, การทดสอบระบบ
10. ข้อจำกัดและความเสี่ยงตามหัวข้อที่ 7 ด้านบน
11. แผนงานและการแบ่งงานทีมTimeline, ผู้รับผิดชอบแต่ละเฟส, milestone
12. ภาคผนวกเกณฑ์ให้คะแนนฉบับเต็ม, ตัวอย่าง prompt, แบบสอบถาม, ภาพหน้าจอระบบ

A ภาคผนวก — เกณฑ์ให้คะแนนและเหตุผลรองรับน้ำหนัก

ใช้ตอบอาจารย์เมื่อถูกถามว่าทำไมถ่วงน้ำหนักแบบนี้

ด้านน้ำหนักเหตุผล
การจ่ายยา (drug)40%การเลือกยาถูกต้อง/ปลอดภัยคือ core competency และผลลัพธ์ด้านความปลอดภัยหลัก
การซักประวัติ (history/WWHAM)25%เป็นฐานสำคัญของการตัดสินใจ แต่เป็น "วิธีการ" ไม่ใช่ปลายทาง
การให้คำแนะนำ (counseling)20%จำเป็นต่อการใช้ยาอย่างปลอดภัยและความร่วมมือในการรักษา
การประเมิน + Red flag (diagnosis)15%น้ำหนักต่ำกว่า แต่ข้อ safety-critical (แพ้ยา/ตั้งครรภ์/refer) ถูกบังคับเพิ่มด้วยการหักคะแนนหนัก (CRITICAL)

หมายเหตุ: งานวิจัยด้าน OSCE/OSPE ในเภสัชศาสตร์ไม่มีน้ำหนักมาตรฐานตายตัว แต่ละสถาบันต่างกัน แนวโน้มคือให้น้ำหนักผลลัพธ์เชิงคลินิกและความปลอดภัยสูง โดยมี WWHAM เป็นฐาน (อ้างอิง: BMC Medical Education OSCE pilot, PMC10091678; งานวิจัยความสัมพันธ์ WWHAM score กับผลลัพธ์ที่สอดคล้องแนวทาง)
